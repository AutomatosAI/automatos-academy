# 08 — Design red-team

**Status:** Review — Week 1. Part of [00-INDEX](00-INDEX.md).

## 0. What this is

An adversarial pass over the whole Mobile Tutor design package (00–07) through four lenses — **Architect, Security, Product, UX** — before any code. The brief was to *try to break it*, not to bless it. Where I make a claim about the academy engine or content schema, I verified it against the live source (`public/js/store.js`, `public/js/engine/readiness.js`, `public/js/views/pathfinder.js`, `public/content/manifest.json`, and the shipped `track.json` + domain files), reconciled against the S03 grounding notes.

**Headline:** the package is genuinely strong — coherent tiers, honest product voice, a real safety boundary, unusually disciplined cross-referencing. But it rests on **one load-bearing factual error that recurs in five docs**: the academy content schema has **no `levels` and no `learning paths`** as first-class objects, and `track.json` has **no inline `domains` array** — yet D3, 01 §7, 02 §2/§3, 03, and 05 SC1 all build on "mirror the academy's levels + paths" and a per-`path` mastery scope. That is not a nitpick; it changes the data model, the selector, and the onboarding screen. Second, **03 §4's readiness gate invents thresholds the engine doesn't have** (an 85% per-domain floor; a 720 pass-mark gate) and contradicts 03 §1's own "reuse `readiness.js`" claim. Neither is fatal — both are resolvable in design — but they must be resolved *before* Claude Design + build, because they ripple into schema, selector, and screens.

## 3. Gap list — what a build agent hits that no doc answers

1. **What *is* a "path"?** No schema, no source data, no rollup input (F1). A build agent cannot build the SC1 "pick a path" branch or the `mastery_map` `path` scope from any existing content.
2. **The exact readiness threshold** (F2) — 0.85 floor or 0.9 A+? Per-domain gate or aggregate? A build agent has two contradictory specs in the same doc.
3. **The decay function** (F5) — column + verb, no algorithm. Curve, half-life, apply-on-read vs write, READY→NOT-READY transition rule: all unspecified.
4. **Track/vendor namespacing of ids** (F4) — item/domain ids collide across tracks; no doc says how the consumer DB disambiguates.
5. **Where mock-exam and scenario state live** server-side (F6/F17) — the gate and bucket-4 need them; 02 §3 has no table.
6. **"Behind plan" definition** (F11) — the selector gates on it; no plan/pace model exists.
7. **First-run prefetch policy** (F16) — what/how-much gets cached on scope-selection so the first offline session works.
8. **Delta-sync conflict resolution beyond "last-write per item"** — 02 §5 says "last-write per item," but two devices answering the *same* item offline, or a `mastery_map` rollup computed on stale local data then reconciled, needs a defined merge (esp. with decay in the mix). Under-specified.
9. **News-card grounding source** (F7) — how a current-events card passes an allowlist of vendor-official docs.
10. **Voice Tutor rate-limit / cost ceiling** (F13) — an open-ended LLM endpoint with no stated per-user bound.
11. **Age-gate placement** (F15) and **additive-scope re-entry flow** (F9) — required/personified but not in any flow diagram.
12. **The product name** (O3) — cosmetic, but Claude Design work and App-Store listing both need it and it's still a placeholder.
13. **KG provenance for the Voice Tutor** — 01/06 say the tutor is "KG-grounded," 04 §4.3 builds a source→content dependency graph, but *no doc defines the KG's schema or how the tutor queries it at runtime* (offline it's dead; online, what's the retrieval contract?). The Voice Tutor's actual grounding mechanism is a black box across the package.

## 5. Verdict

### **GO-WITH-FIXES.**

This is a well-constructed package. The three-tier model holds, the draft→published safety boundary is genuinely well-reasoned (04/06 are the strongest docs — the allowlist-as-supply-chain-boundary and factory-observability sections are excellent and grounded in a real prior incident), the product voice is honest and consistent, and the cross-referencing discipline (every doc conforms-checks D1–D5, hands off cleanly, flags its own open questions) is better than most shipped PRDs. The reframe — "70% delivery of engines you already own" — is *directionally* right and the SM-2 half of it is verified true.

But it is **not** GO as-is, because it rests on a factual overstatement of what the academy content model contains, and that overstatement has already propagated into the data model, the selector, and the onboarding screen. Design that builds on data that doesn't exist will dead-end at the first `GET /catalog/{track}` that returns no `paths`.

**None of the fixes are architectural rewrites** — they're reconciliations and specifications, all resolvable inside this design phase. So: proceed to Claude Design + build **after** the must-fix shortlist is closed.

### Must-fix-first (blocks Stage 0 exit / the Content API sign-off)

1. **F1 / D-01 — define "path" and "level" against reality** (redefine over tracks, or commission as new schema). Everything downstream inherits this. *Highest leverage single fix.*
2. **F3 — correct the 02 §2 Content API contract table** to the real `manifest`/`track.json`(`domainFiles`)/domain-file shape. It's the literal Stage-0 exit artifact and it's currently wrong.
3. **F2 / D-02 — settle the readiness gate** (one definition; label new-vs-reused honestly). It's contradicted three ways and it's the product's headline promise.
